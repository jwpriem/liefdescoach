## 2024-04-16 - [Fix User Enumeration in OTP Request]
**Vulnerability:** The `/api/auth/send-otp.post.ts` endpoint failed to record an email in the `otpRequestsByEmail` rate-limiting map when the email didn't exist in the database. This allowed attackers to send two requests for an email in rapid succession: if the second request was rate-limited (429), the email existed; if it wasn't, the email did not exist.
**Learning:** Returning a generic response (like `{ success: true }`) is insufficient if the rate-limiting state differs based on the user's existence. Attackers can leverage these behavioral differences (in this case, whether the rate limiter triggers) to map out existing accounts.
**Prevention:** Always ensure that behavioral state changes (like incrementing rate limiters) are applied uniformly before any logic branches based on whether the entity (e.g., user account) exists in the database.
## 2024-05-18 - [Fix Email Spamming in Password Reset Endpoint]
**Vulnerability:** The `/api/auth/request-password-reset.post.ts` endpoint lacked a resource-specific (email) rate limit. While it properly rate-limited requests by IP (e.g. 5 requests per hour), an attacker rotating IP addresses could infinitely spam password reset emails to the same target email address, causing an application-layer Denial of Service (DoS) and in-box flooding.
**Learning:** Depending solely on IP rate limiting for external side-effect endpoints (like sending emails or SMS) leaves systems vulnerable to distributed abuse. Both IP-based (broad) and resource-based (narrow) rate limits are required.
**Prevention:** Implement resource-specific rate limits (like email addresses or phone numbers) alongside IP rate limits. To maintain user enumeration protections, ensure these resource limits trigger silently (returning a generic success response) prior to performing database lookups.
## 2024-05-24 - [Fix SMTP Floating Promises in Endpoints]
**Vulnerability:** In `server/api/mail/send.post.ts` and `server/api/auth/request-verification.post.ts`, the external `sendMail` calls were either handled as synchronously blocking calls or as "floating promises" (`.catch(...)` without `await` or `waitUntil`). This caused two issues: it tied up the main server process when synchronously blocking, presenting a Denial of Service (DoS) risk, and floating promises could be prematurely terminated in serverless environments, leading to silent email drops.
**Learning:** Sending emails inline without correctly detaching them from the HTTP request context is both a performance bottleneck and a reliability hazard. Relying on floating promises is dangerous because the runtime might suspend the process as soon as the HTTP response is sent.
**Prevention:** Always defer conditionally executed tasks (like SMTP `sendMail`) to the background by wrapping them in `event.waitUntil(Promise.resolve().then(async () => { ... }))` to guarantee the HTTP response is returned immediately in constant time while ensuring the background task runs reliably.
